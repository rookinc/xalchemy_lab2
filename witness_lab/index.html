Harness README

This page is the Sprint 1 witness harness. It is intentionally console-first. The goal is to validate the machine, the operators, and the readout before any graphics layer is allowed into the stack.

The harness is built around the real witness machine, not a placeholder browser simulator. The Python engine is the source of truth. The browser acts only as a transport surface and inspection console.

What this page does

It exposes a live readout of a current witness state and lets you apply the canonical operators one step at a time or on autoplay. The console shows the state payload returned by the API after each transition.

Operators

τ advances the frame while keeping phase fixed. τ⁻¹ reverses the frame step while keeping phase fixed. μ flips the phase while preserving the frame.

Current Sprint 1 contract

  • The witness machine is canonical.
  • The API is a thin transport layer.
  • The harness is a readout and control surface only.
  • No rendering claims are made yet.
  • The console output is the primary debugging instrument.

Readout fields

The console reports the current state code, frame, phase, phase label, witness cycle, species, alignment, spread, fiber, action cell, action species, and the immediate τ, τ⁻¹, and μ targets. It also keeps a short recent history trace so local temporal behavior is visible at a glance.

What to look for

  • Under repeated τ, frame should cycle while phase remains fixed.
  • Under repeated μ, phase should alternate subjective/objective.
  • Alignment, spread, and fiber should toggle with phase.
  • The recent history trace should reveal local orbit structure.

Why no visuals yet

Because Sprint 1 is about trusting the numbers. Once the state transitions and readouts are stable, later sprints can map the same machine payload into visual geometry without changing the source of truth.
